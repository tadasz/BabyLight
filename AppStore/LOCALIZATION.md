# Baby Light — 15-language localization & v1.1 staging

This documents the localization pass (app + App Store listing) into the 15 biggest
App Store languages, and exactly how to ship it as **version 1.1**.

## Languages (16 total: English + 15)

`en-US` (source) · `zh-Hans` · `zh-Hant` · `ja` · `ko` · `ru` · `de-DE` · `fr-FR`
· `es-ES` · `es-MX` · `it` · `pt-BR` · `nl-NL` · `tr` · `pl` · `ar-SA`

These cover the largest App Store markets (incl. both Chinese variants and both
Spanish variants). Arabic (`ar-SA`) is right-to-left.

## What was done

### 1. The app itself is localized (in the binary)
- `Baby Light/Localizable.xcstrings` and `Baby Light Watch App/Localizable.xcstrings`
  are String Catalogs with all UI strings in 15 languages.
- Swift was made localizable: model-provided strings now resolve through
  `LocalizedStringKey` (`ControlsOverlay.swift`, `WatchContentView.swift`).
- `knownRegions` in the project lists all 16 locales.
- The Release archive bakes in `ar/ de/ es/ es-MX/ fr/ it/ ja/ ko/ nl/ pl/ pt-BR/
  ru/ tr/ zh-Hans/ zh-Hant/ en` `.lproj` resources → the App Store will list all 16
  languages, and the app runs localized on-device.

### 2. The App Store listing is translated (all 16 locales)
- Single source of truth: the **registry** `AppStore/i18n/locales.json` + the English
  `AppStore/i18n/source.json` + one file per language `AppStore/i18n/translations/<asc>.json`
  (name, subtitle, promo, keywords, description, what's-new, screenshot captions, and
  the in-app UI strings). Every field is validated against Apple's character limits by
  `make validate`. **Adding a language is config-only — see [`ADD_A_LANGUAGE.md`](ADD_A_LANGUAGE.md).**

### 3. Localized screenshots
- `AppStore/screenshots/loc/<locale>/01..06.png` — 90 iPhone 6.5" creatives
  (1284×2778), captions + in-app control labels translated, script-correct fonts,
  Arabic shaped + right-to-left. Generated by `AppStore/i18n/screenshots.py`
  (fonts chosen by each locale's `script` in the registry).
- iPad (12.9") and Apple Watch sets stay English; App Store falls back to the
  primary-language screenshots for those device families per locale.

### 4. The build (Xcode Cloud)
- The localized **v1.1** build is produced by **Xcode Cloud** (Apple-managed cloud
  signing — no local certificates needed), using the existing **"AppStore"** workflow
  (`8c13de8a-…`, APP_STORE_ELIGIBLE archive). Because that workflow is pinned to
  `main`, on-demand builds of a feature branch are started with `asc/xcode_cloud.py`,
  which temporarily adds the branch to the workflow's start condition, starts a manual
  `ciBuildRun`, then restores the workflow. Trigger + watch + report:
  ```bash
  cd AppStore/asc
  python3 xcode_cloud.py build-for claude/happy-yalow-d38b66          # App Store build
  # python3 xcode_cloud.py build-for <branch> --testflight           # internal TestFlight build
  ```
- **Build numbering:** Xcode Cloud stamps `CFBundleVersion` = the run number at
  archive time (it overrides `CURRENT_PROJECT_VERSION`, so editing the project in a CI
  script has no effect — verified). Run numbers only increase and are already past the
  one-off local stopgap build **1.1(14)**, so the Xcode Cloud build comes out at
  **1.1(15)**, which App Store Connect accepts (higher than any prior build).
- **Fallback (local):** a local archive can still be made and uploaded with
  `asc/ExportOptions.plist` (manual signing) + `xcrun altool --apiKey`; App Store
  provisioning profiles for that path are created by `asc/create_profiles.py`. This is
  only a fallback — prefer Xcode Cloud.

## Why it isn't submitted yet

Version **1.0 is still in Apple's review queue** (first release, English only).
Apple's API refuses to create or edit a second version until 1.0 is released:
`"You cannot create a new version of the App in the current state."`

Per the chosen plan, 1.0 ships as-is, and **everything above is staged** to go out
as 1.1 the moment 1.0 is approved.

## How to ship 1.1 (one command, after 1.0 is approved)

```bash
cd AppStore
make stage          # creates 1.1, pushes every locale + screenshots, attaches the latest build
make submit         # …and also submits 1.1 for review
```

`stage_v11.py` (what `make stage` runs) is idempotent and self-checks: if 1.0 isn't
released yet it prints a clear message and exits without changes.

## Everyday commands (`make` from `AppStore/`)
| command | what it does |
|---|---|
| `make validate` | check every locale for completeness + Apple char limits |
| `make catalogs` | regenerate the iOS + watch String Catalogs |
| `make screenshots` | render localized screenshots (`LOCALES="de-DE fr-FR"` for a subset) |
| `make all` | validate + catalogs + screenshots |
| `make add-locale ASC=.. XCODE=.. SCRIPT=.. NAME=..` | scaffold a new language |
| `make build` | build the current branch via Xcode Cloud (App Store) |
| `make stage` / `make submit` | stage / submit the next App Store version |

## Files
**`AppStore/i18n/`** — localization pipeline:
`locales.json` (registry) · `source.json` (English) · `translations/<asc>.json` ·
`lib.py` (shared loader+validator) · `build_catalogs.py` · `screenshots.py` ·
`scaffold_locale.py`.

**`AppStore/asc/`** — App Store Connect API:
| file | purpose |
|---|---|
| `asc.py` | tiny ASC API client (JWT via `AuthKey_DX75UNTZ4U.p8`) |
| `xcode_cloud.py` | start/poll Xcode Cloud build runs (primary build path) |
| `push_metadata.py` | create/update appInfo + version localizations (registry-driven) |
| `upload_screenshots.py` | upload the localized iPhone 6.5" screenshot sets |
| `stage_v11.py` | orchestrate the next-version staging |
| `create_profiles.py` / `ExportOptions.plist` | local archive+altool **fallback** signing |

Credentials: the API key id (`DX75UNTZ4U`) and issuer are defaults in `asc.py`; the
private key stays at `~/.appstoreconnect/private_keys/AuthKey_DX75UNTZ4U.p8` and is
**not** committed.
